# Here's a walkthrough of what the app actually does, feature by feature.

The local/remote toggle is the piece that solves your "don't run it on local CPU" concern. The Streamlit app never assumes where the model lives — it talks to a small abstraction layer (utils/inference.py) that exposes one predict() method with two implementations behind it. In local mode, that method loads best.pt directly into the Streamlit process using Ultralytics and runs inference right there, which is simple but ties up whatever machine is running the app. In remote mode, the same method instead JPEG-encodes the current frame and POSTs it to a separate FastAPI server (server/app.py), which you deploy independently — for free, on something like a Hugging Face Space with the Docker SDK — and which does the actual model inference and tracking. Switching between the two is a sidebar radio button, not a code change, so you can develop against a local GPU box and demo against a free hosted instance without touching anything.

The configuration sidebar exists because tuning a detection pipeline is mostly about finding the right balance between speed and accuracy for your specific footage, and that balance is never the same across cameras, lighting, or hardware. Confidence threshold controls how sure the model needs to be before it draws a box at all — raise it and noisy borderline detections disappear, but so do some real, low-confidence ones. The IoU/NMS slider controls how aggressively overlapping boxes on the same object get merged into one; too low and you get duplicate boxes on a single person, too high and you might collapse two people standing close together into one box. Resolution and target FPS are the resource knobs — smaller frames and fewer frames processed per second mean less CPU/GPU load, at some cost to how well small or fast-moving objects get picked up. None of these are things you can pick correctly in the abstract; they depend on your camera, your object sizes, and your hardware, which is why the sliders exist as live, adjustable controls rather than hardcoded constants.

The benchmark tab is there so you don't have to guess at that speed/accuracy tradeoff — it grabs a handful of frames from whatever source is currently configured and actually runs your model against each resolution preset (320 up to 1280), timing the latency and counting detections at each one. The two bar charts it produces show you, concretely, "if I drop to 480px I get roughly this much more FPS, and here's whether the model still finds the same number of objects." It's an honest tool, not a rigorous one — the detection count is a stability proxy, not real accuracy, since there's no ground-truth labels to check against. But it's enough to make an informed choice about where to set the resolution/FPS sliders before running the app on a live feed.

Temporal filtering and ByteTrack work together to cut down false positives, which was one of your original asks. ByteTrack assigns a persistent ID to each detected object across frames, so the same person or hard hat keeps the same ID as long as they stay roughly in view, even through a frame or two of occlusion. The temporal filter then uses those IDs to enforce a rule: a detection only gets drawn and counted once it has shown up under the same ID for N consecutive frames (the slider default is 5). A real object sticks around across frames, so it clears that bar easily. A one-off false positive — a shadow, a reflection, a weird lighting flicker — typically appears for a single frame and vanishes, so it never accumulates enough frames to get confirmed, and it never reaches your screen or your alert log.

The zone classification feature is the newest piece, built off your sample image. Every detected person box gets split into three equal vertical bands, and the split is computed from that specific person's actual box height — not a fixed assumption about how tall a person "should" be in frame — so someone who's only half-visible because they're cropped by the edge of the camera still gets three bands scaled to whatever's actually visible. There's also a sanity check on the box's proportions: if a box is too short and wide to plausibly be a standing person, the zones get generic labels ("zone 1/2/3") instead of confidently calling them "head/torso/legs" on something that might not even be a full body. Once the bands exist, every PPE detection (helmet, vest, boots, etc.) that falls within a person's horizontal extent gets assigned to whichever band its center point lands in, and matched to the correct person when there's more than one in frame.

The practical payoff of the zone system is that it doubles as another false-positive filter, on top of temporal filtering and ByteTrack. Each PPE class has an expected zone — helmets belong near the top, boots near the bottom — so when a detection lands somewhere it has no business being (a "helmet" near someone's feet, say), the app flags it in red with a (!) marker instead of treating it as a normal, trustworthy detection. That flag is visible both on the drawn boxes and as a running count in the stats panel, so you can see at a glance how often the model is producing geometrically implausible detections, which is often a good signal that something needs retraining or that the confidence threshold needs to come up.
